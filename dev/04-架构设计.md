# 架构设计（易读版）

## 1. 五层结构

```text
UI：窗口、Controller、Editor、曲线/三维控件
  ↓
Service：打开、保存、建模、建工况、导出、计算、查结果
  ↓
Model：ProjectDocument、领域对象、ComputeCase、结果对象
  ↓
适配器：sopro codec、INPUT exporter、OUTPUT parser
  ↓
外部资源：.sopro、INPUT、OUTPUT、SolverRunner
```

每层只做自己的事：

- UI 负责收集输入和显示，不解析 XML、不拼 INPUT、不直接启动求解器；
- Service 负责组织一次完整操作；
- Model 只保存数据和 ID 关系，不保存控件、进程或大型结果数组；
- 适配器负责文件格式转换；
- SolverRunner 负责进程、日志、退出码和取消。

## 2. 一条完整调用链

```text
打开 .sopro
  → ProjectStorage/loader
  → ProjectDocument 和 ProjectNode
  → Editor 修改领域模型
  → CaseService 创建 ComputeCase
  → CalculationService 调用各领域 exporter
  → INPUT → SolverRunner → OUTPUT
```

结果和三维的后续链路已经预留，但负责人未定：

```text
OUTPUT → ResultCatalog → ResultDataset/CurveSeries/MatrixData
       → 结果 UI 或 SceneModel/SceneFrame
```

## 3. 公共边界

跨模块只通过稳定 ID、公开方法和公共对象调用：

- 工程：`ProjectDocument`、`ProjectNode`、`ResourceRef`；
- 错误：`ValidationIssue`、`CommandResult`；
- 计算：`ComputeCase`、`CasePaths`、`ExportReport`；
- 结果（待分配）：`ResultDataset`、`CurveSeries`、`MatrixData`、`SceneModel`、`SceneFrame`；
- UI 注册：`EditorRegistry`。

不要在另一个负责人的模块中直接改内部字段；需要增加公共字段时先记录合同变更。

## 4. 责任关系

| 链路 | 负责人 | 交给谁使用 |
|---|---|---|
| 工程、环境、海域 | A | B/C 读取模型和资源引用 |
| 浮体、水动力、工况、计算 | B | A 保存工程，C 使用 `case_id` 和 OUTPUT 入口 |
| 连接、管缆、管缆 INPUT | C | B 的计算调度调用 exporter |
| 结果、曲线、三维 | 待分配 | B 提供 OUTPUT，A 提供环境/工程数据 |

## 6. 文件负责人速查

| 负责人 | 主要维护文件 | 简单理解 |
|---|---|---|
| A | `core/models/project.py`、`node.py`、`resource.py`、`validation.py`；`core/sopro/*` 的工程/环境文件；`app/services/project_service.py`；工程 Controller 和 `EditorRegistry` | 让工程能打开、编辑、保存和重新打开 |
| B | `core/models/case.py`、浮体/水动力模型；`app/services/case_service.py`、`calculation_service.py`；`core/solver/*`；计算 Controller/Worker | 让模型生成 INPUT，运行求解器并归档 OUTPUT |
| C | 连接/管缆模型；`connection_codec.py`、`cable_codec.py`；管缆 exporter 和 Editor | 让连接和管缆能保存并写入 INPUT |
| 待分配 | `core/results/*`、`result_service.py`、结果 Controller/图表/矩阵/三维控件 | 让 OUTPUT 变成曲线、矩阵和静态/动态三维 |

共享文件只通过公共合同使用。一个文件只有一个主要维护者；结果/三维负责人确定前，不要提前实现或抢改“待分配”目录。

## 7. 当前代码与目标的关系

公共模型、工况、结果合同、EditorRegistry 和跨模块 fake 测试已经存在；完整 Service、真实 codec/exporter、OUTPUT parser、结果 UI 和三维仍是后续任务。文档中的目标文件不代表它们已经全部实现。
